## Step 0 ‚Äî Context, source-of-truth limitation, and where business logic lives

**No PRD / requirements doc exists.** bitwarden/server is an open-source product backend; there
is no `prd.md`, requirements spec, or acceptance-criteria doc in the repo. The authoritative
sources for the domain are therefore:

1. **`README.md`** ‚Äî one paragraph of product framing: "The Bitwarden Server project contains the
   APIs, database, and other core infrastructure items needed for the 'backend' of all bitwarden
   client applications." (`README.md:13`); "The database is written in T-SQL/SQL Server."
   (`README.md:14`). External help-center/contributing docs are _linked_ but live outside the repo.
2. **The code itself** ‚Äî entities, enums, XML-doc comments, command/service names, stored procs.
   Notably, several enums carry rich domain docstrings (e.g. `OrganizationUserStatusType.cs:5-33`)
   that function as the _de facto_ requirements text.

**Consequence for this artifact:** the Step-4 mismatch table is **code-reality-driven, not
doc-vs-code.** There is no document to contradict the code; the "model" side of each mismatch is
the _conceptual model implied by the domain language and the code's own docstrings_, and the "code"
side is what the implementation actually does. This is a weaker form of mismatch detection than a
true doc-vs-code diff ‚Äî called out explicitly as a limitation.

**Stack & layers (where business logic lives).** Confirmed via the L2 map and the lifecycle trace:

| Layer                                                                    | Role                                                                    | Example (verified)                                                                                                                                                  |
| ------------------------------------------------------------------------ | ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Hosts** (`Api`, `Identity`, `Scim`, `Admin`, ‚Ä¶)                        | Thin controllers + config; no domain rules                              | `Api/AdminConsole/Controllers/OrganizationUsersController.cs` (thin actions delegating to Core)                                                                     |
| **Core ‚Äî Services / Commands / Validators**                              | **The domain layer.** All invariants live here                          | `Core/AdminConsole/Services/Implementations/OrganizationService.cs`; `Core/AdminConsole/OrganizationFeatures/OrganizationUsers/**` (commands + validator pipelines) |
| **Core ‚Äî Entities / Enums**                                              | Anemic data holders + domain vocabulary (docstrings)                    | `Core/AdminConsole/Entities/OrganizationUser.cs`, `‚Ä¶/Enums/*.cs`                                                                                                    |
| **Infrastructure** (`Dapper`+`Sql/dbo` stored procs / `EntityFramework`) | Persistence only; runtime-selected by `GlobalSettings.DatabaseProvider` | dual stack, switch at `SharedWeb/Utilities/ServiceCollectionExtensions.cs:98-124` (per L2/research)                                                                 |

Business logic concentrates in **Core services and the newer `OrganizationFeatures/**` command +
validator pipelines**. Entities are **anemic\** (data + a few helpers); they do *not\* guard their own
invariants ‚Äî a central finding for Steps 3‚Äì5.

## Step 2 ‚Äî Subdomain classification (Core / Supporting / Generic)

What makes Bitwarden valuable is the **encrypted secrets store** (vault) and the **zero-knowledge
key exchange** ‚Äî that is what users pay for and what no generic component can replace. Org/member
management is what makes it _sellable to businesses_ (the revenue driver) but is competitively
ordinary; billing and email are commodity.

| Subdomain                                         | Class                              | Justification (evidence-based)                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------- | ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Vault / Secrets** (encrypted items)             | **CORE**                           | The product's reason to exist ‚Äî README frames the server as backend "for all bitwarden client applications" (`README.md:13`); L2 map lists `Core/Vault` + `Core/KeyManagement` as deep modules. The encrypted store is the differentiator. _(Outside this artifact's focus; classified for context.)_                                                                                                           |
| **Key management / cryptographic key exchange**   | **CORE**                           | Zero-knowledge is the moat. Surfaces in _this_ area as the Confirm step: org symmetric key encrypted with the member's public key (`OrganizationUser.Key` `:40-44`, set on Confirm). Getting this wrong breaks the security guarantee ‚Äî irreplaceable.                                                                                                                                                          |
| **Organization & member management** (this focus) | **CORE‚Äìleaning SUPPORTING**        | It is the **revenue engine** (orgs = paid tenants; seats = the billing unit) and the busiest module in the repo (L2: `Core/AdminConsole` #1 every quarter, 1,048 commits). It directly mediates the core asset (the Confirm key exchange lives here). Classified **Supporting with a Core seam**: the _management_ (invite/seat/role workflow) is supporting; the _key-exchange-on-confirm_ it carries is Core. |
| **Policies**                                      | **SUPPORTING**                     | Enterprise differentiator (2FA enforcement, SSO, single-org). 22 policy types (`PolicyType.cs`) directly gate membership transitions. Valuable to the org-management value prop but not the cryptographic core.                                                                                                                                                                                                 |
| **Provider / MSP**                                | **SUPPORTING**                     | Lets managed-service providers administer client orgs; extends the org-management value to a channel segment. Distinct entity family (`Entities/Provider/*`, `Enums/Provider/*`).                                                                                                                                                                                                                               |
| **Billing / Subscriptions / Seats**               | **SUPPORTING**                     | Monetizes orgs (seats, plans). Deep module (L2: `Core/Billing` #2). Bitwarden-specific _plans_ but mechanically commodity; Stripe-facing. Seat math is woven into invite/restore (Step 3).                                                                                                                                                                                                                      |
| **Identity / Auth / SSO**                         | **SUPPORTING (security-critical)** | Gatekeeper to everything; security-critical and bus-factor-risky (L2 ¬ß4). Not the secrets-store core, but enables it.                                                                                                                                                                                                                                                                                           |
| **Email / notifications / push**                  | **GENERIC**                        | Commodity delivery. `IMailService`/`IPushNotificationService` seams with swappable impls (research: 5 `IMailDeliveryService` impls config-selected). Replaceable with off-the-shelf.                                                                                                                                                                                                                            |
| **Events / audit**                                | **GENERIC‚ÄìSUPPORTING**             | Audit log (`IEventService`). Generic plumbing; mildly supporting for enterprise compliance.                                                                                                                                                                                                                                                                                                                     |
| **Data-access (Dapper/EF/SQL dual stack)**        | **GENERIC (infrastructure)**       | Persistence mechanism, not domain. Essential-complexity per `_intent.md` (cloud-vs-self-host matrix) but carries no business meaning.                                                                                                                                                                                                                                                                           |
