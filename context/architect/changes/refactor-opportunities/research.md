### Lens c — MIGRATION FEASIBILITY (incremental, reversible path)

**Adequate target shape (naming only, per limits):** all org-member-invite surfaces route through `InviteOrganizationUsersCommand`; legacy `OrganizationService.InviteUserAsync`/`InviteUsersAsync`/`SaveUsersSendInvitesAsync` deleted; both flags retired. Per-surface request adapters map each caller's model to `InviteOrganizationUsersRequest`.

**Existing vs new abstraction:**

- The receiving abstraction **already exists and is production-proven** (command + validator pipeline + shared private core `:114`). **[evidence]**
- What's missing is **one thin public entry point + a request adapter per remaining surface** (web, import, provider). The pattern to copy is exactly `InviteImportedOrganizationUsersAsync` (`:80`) — a ~30-line public wrapper over the private core differing only in response mapping. **[evidence — inference that web needs the same shape]**
- This is therefore an **incremental, reversible** path: add a flag-gated `_vNext` branch per surface (mirroring `MembersController.cs:173-210`), bake, then delete legacy. Reversible at each step by flag-off. **[inference, grounded in the existing MembersController pattern]**

**Blast-radius data (from priors + this pass):**

- Co-change (research verified #13, since 2024-12): `InviteOrganizationUsersCommand.cs` 11, `SendOrganizationInvitesCommand.cs` 9, `InviteOrganizationUserValidator.cs` 8 — a rule change ripples validator → error type → mapper. **[evidence]**
- A _data-write_ change is the proven **5-way atomic change** (Dapper + EF + proc + migrator + parity test; research verified #10, `4de10c83`). **BUT completing this migration is mostly a Core/Api control-flow change, not a write-method change** — it reuses existing repository writes, so it should _not_ trigger the 5-way tax unless a write signature changes. **[inference — the new command calls the same `IOrganizationUserRepository` seam; research verified #3 "every write goes through the interface"]**
- Surfaces touched per increment: 1 controller/command + 1 request-model adapter + flag check. Web increment additionally has **no existing controller-level vNext scaffold** (unlike MembersController), so it is the heaviest of the three. **[evidence]**

**Guards/tests that already exist (CI-confirmed — resolves a prior `unknown`):**

- **The Dapper↔EF parity guard IS enforced in CI.** `.github/workflows/test-database.yml` spins up MSSQL+Postgres+MySQL+MariaDB+SQLite and runs `test/Infrastructure.IntegrationTest` against **all five** (`BW_TEST_DATABASES__0..4`), triggered on any change under `src/Sql`, `util/*Migrations`, `Infrastructure.Dapper`, `Infrastructure.EntityFramework`, or `**/Entities/**`. **[evidence — workflow lines 11-31, 121-138]** This **resolves** the prior `unknown` ("test-runner's enabled providers… not statically inspectable"): in CI all five are configured. The `[CiSkippedTheory]` `EFIntegration.Test` suite (research verified #17) is a _separate, narrower_ suite; the load-bearing parity guard is `Infrastructure.IntegrationTest`, which does run. **[evidence + inference]**
- Unit coverage exists for both lineages: legacy `OrganizationServiceTests.cs` (52 `[Theory]`), new `InviteOrganizationUserCommandTests` (14) (research verified #20). **[evidence per prior]**
- **Gap that bites this migration:** no e2e invite→accept→confirm sequence test (research item 4 / verified #16). Completing the web cutover changes the invite half of that chain with **no single end-to-end guard** that the web invite still produces a valid invitee that can accept+confirm. **[evidence — this is finding F, now reclassified as the key prerequisite test for A]**

**First prerequisite step (no code change — exploration verdict):** before any cutover, **establish the behavioral-parity guard for the web/AdminConsole path**: an integration test (or characterization test) asserting that the legacy `OrganizationUsersController` invite and a candidate new-command web entry point produce identical observable outcomes (member rows at `Status=Invited`, same emails/events/billing side-effects). This is the missing safety net that makes the otherwise-mechanical "copy the `InviteImported` wrapper for web" reversible-and-safe. Until that guard exists, completing the migration is not de-risked. **[inference, grounded in F + the existing test-database CI harness]**

## Refactor opportunities (ranked)

Honest count: **one** genuine structural candidate. Ranking is a PROPOSAL for a later planning session, not a decision.

### 1. Complete the invite-command Strangler Fig migration _(the only candidate)_

- **Current → target shape:** Two parallel invite lineages (legacy 1076-line `OrganizationService.InviteUsersAsync` god-method serving web/import/provider unflagged + two half-migrated flag-gated surfaces) → single `InviteOrganizationUsersCommand` (already-built command + validator pipeline) serving **all** org-member-invite surfaces, with legacy methods and both flags deleted.
- **Why it ranks #1 (debt cost vs change cost):** It is the **only** finding whose fix changes code structure rather than being a test/config/process item or settled essential complexity. **Debt cost** is real but bounded and _rising_: every invite change today must be applied to two lineages (research: "a fix to one path can diverge behavior between internal and public APIs"), three surfaces still have no vNext at all, and there is **no cutover/cleanup commit** — \_intent's explicit "flips toward ACCIDENTAL if activity stops" risk. **Change cost** is moderate and front-loaded by the missing parity guard, but mechanically low: the target abstraction exists and is production-proven, the per-surface increment mirrors the existing `MembersController` switch + `InviteImported` wrapper, and it largely avoids the 5-way data-write tax.
- **Blast radius:** Per increment — 1 controller/command + 1 request-model adapter + 1 flag check; reuses existing repository writes (no expected proc/EF/migrator churn unless a write signature changes). Co-change hotspots if validation rules are touched: command (11) / validator (8) / error mapper. Web increment is heaviest (no existing controller scaffold). **[evidence/inference as cited in Lens c]**
- **Incremental path sketch (reversible at each step):** (1) build behavioral-parity guard for web invite; (2) add flag-gated `_vNext` web entry point on the command (copy `InviteImported` wrapper + a web request adapter), bake; (3) repeat for import, then provider; (4) once all surfaces are on `_vNext` and baked, delete legacy `InviteUserAsync`/`InviteUsersAsync`/`SaveUsersSendInvitesAsync` and retire `PublicMembersInviteRefactor` + `ScimInviteUserOptimization`. Each step is flag-reversible.
- **First prerequisite step:** establish the web-path behavioral-parity guard (see Lens c) — _before_ writing any cutover code. Everything else is gated on having that safety net.

_(No #2 or #3. Manufacturing additional candidates would violate the brief — the pool is genuinely this small once essential complexity and non-structural items are removed.)_
